# Output Control Rules

Use these rules to control output size and reduce wasted tokens.

## Output Modes

### Compact

Use for:
- Quick answers
- Claude Code handoffs
- File creation instructions
- Simple edits
- Summaries
- Confirmations

Format:
- Short sections
- Tables when useful
- No long explanations
- Only necessary details

### Standard

Use for:
- Normal strategy work
- Page outlines
- Keyword maps
- Gap fixes
- Build briefs
- QA summaries

Format:
- Clear sections
- Prioritized recommendations
- Enough detail to act
- Avoid unnecessary theory

### Deep

Use for:
- Competitive analysis
- Full alignment reviews
- Prompt evaluation
- Launch QA
- High-value page strategy

Format:
- Scorecards
- Matrices
- Detailed reasoning
- Risks
- Priorities
- Implementation notes

## Output Rules by Task

Keyword strategy:
Use standard output unless the page is highly competitive.

Page outline:
Use standard output for Core Mode, deep output for Beyond-Elite Mode.

10-metric analysis:
Use deep output for important pages, standard output for normal pages.

Visibility alignment:
Use deep output only for important pages.

Developer build brief:
Use compact to standard output.

Claude Code build prompt:
Use compact, implementation-focused output.

QA review:
Use standard output for normal pages, deep output for launch pages.

Prompt evaluation:
Use deep output.

## Avoid

- Repeating the full page brief
- Repeating previous outputs unless requested
- Writing full content before approval
- Giving every possible recommendation
- Adding theory when the user needs action
- Creating long outputs for low-priority tasks

## Preferred Formats

Use:

- Tables
- Scorecards
- Priority lists
- Pass/fail labels
- Required/recommended/optional labels
- Implementation notes
- Next-step summaries

Avoid:

- Long essays
- Repetitive explanations
- Generic SEO commentary
- Full rewrites unless requested

## Final Rule

Match output length to task value.
